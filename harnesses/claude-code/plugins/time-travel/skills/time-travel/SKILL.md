# Time Travel

A multi-agent premortem skill. Sends stakeholder personas to the future, brings back a structured failure analysis, and rewrites the plan with mitigations folded in.

## When to invoke

Activate this skill when the user:

- Types `/time-travel:run` (with or without an argument)
- Asks to "premortem" / "pre-mortem" / "stress test" / "time travel" a plan
- Asks "what could go wrong with this plan?" / "find the failure modes" / "kill this plan if you can"
- Wants an audit of a plan that was generated by an LLM (especially when about to share with leadership)
- Asks for a "skeptic check" or adversarial review of a forward-looking proposal

If unsure whether this skill applies, lean toward invoking — the cost of running is bounded; the cost of missing a real failure mode is not.

## Invocation contract

```
/time-travel:run [source]

source            Optional. One of:
                  - path/to/plan.md            file path
                  - "literal plan text..."     quoted plan text
                  - (omitted)                  auto-detect (see below)

Auto-detect order when source is omitted:
  1. Latest file in ./plans/ (CWD)
  2. Latest file in ~/.claude/plans/
  3. Last substantial assistant message in the current conversation
  4. Prompt the user to paste

Flags:
  --fast            Single-call mode (skip parallel persona dispatch).
                    ~20% of token cost. Lower richness.
  --horizons LIST   Override the default 3mo,6mo,12mo (e.g. --horizons 1mo,6mo,2y).
  --output PATH     Override default output dir (./time-travel-reports/).
  --for-exec        Emit additional ≤300-word exec one-pager artifact.
  --ai-source       Auto-include the AI-Optimism persona and apply tighter
                    rhetorical scrutiny to the plan's wording.
  --no-process-log  Skip the process log artifact.

Env var:
  TIME_TRAVEL_REPORTS_DIR    Override the default output directory globally.
```

## The flow (six phases)

### Phase 0 — Ingest the plan

1. Resolve the plan source using the auto-detect order above.
2. Read the plan. Extract: goal, domain, named stakeholders, timeline, success criteria, key bets, intended audience.
3. Check for AI-optimism triggers. If `--ai-source` is set, OR the plan source matches AI-pattern heuristics (see `references/grounding.md`), flag for AI-Optimism persona inclusion in Phase 2.
4. If parsing yielded too little structure to reason about (no clear goal or no clear timeline), ask the user 1–2 targeted clarifying questions before proceeding. Do not ask more than 2.

### Phase 1 — User-POV pass

Before any adversarial work, capture the user-side reality. Main thread, ~150 words. Answer:

- What did users / customers / beneficiaries think they were getting from this plan?
- What would they forgive if things went sideways?
- What about this plan would feel untrustworthy to them?
- What kinds of complaints would predictably arrive in support / Slack / NPS?

Save these answers — they become a section in the final report and they prime the personas in Phase 3.

### Phase 2 — Roster + evidence

1. **Pick the persona roster.** Select 5–7 personas from `references/persona-library.md`:
   - 2–3 universal personas (always-relevant: skeptical-money, frustrated-end-user, change-resister)
   - 3–4 domain-specific personas chosen to match the plan's industry and stakeholders
   - The AI-Optimism persona if Phase 0 flagged it
   - You may invent a new persona if the plan's domain is exotic and the library doesn't cover it; document the rationale in the process log.

2. **Gather evidence.** Run 3–5 focused web searches using the heuristics in `references/grounding.md`. Look for real-world analogue failures of similar plans. Build a shared evidence pack (~600 tokens) that every persona will see in Phase 3.

### Phase 3 — Parallel persona fan-out

Dispatch each persona as a **separate parallel `Agent` subagent** using the `superpowers:dispatching-parallel-agents` pattern. Each persona receives:

- The full plan
- Its persona definition (POV, biases, what it cares about)
- The shared evidence pack from Phase 2
- The User-POV findings from Phase 1
- The prompt template from `references/timeline-prompts.md`

Each persona returns:

- **Three time-stepped narratives**: a short story at T+3mo, T+6mo, T+12mo describing what it's observing from inside the plan's world at that horizon.
- **3–5 failure modes**, each tagged with:
  - Severity scores across the 6 lenses (`references/severity-lenses.md`)
  - Likelihood (Low / Medium / High)
  - Tripwire — the first observable signal in the field
  - **What would change my mind** — the specific evidence that would reduce confidence in this risk
  - Citation — which evidence-pack item or persona POV supports it

If `--fast` is set: skip the parallel dispatch. Instead, run all personas through a single structured prompt in the main thread. Output quality will be lower but token cost drops sharply.

### Phase 3.5 — Rebuttal round

Synthesize a digest of each persona's top 2 findings (no attribution). Send the digest back to each persona (parallel `Agent` calls again) and ask:

- Where do you agree with the others?
- Where do you disagree, and why?
- Do any of these findings interact — does failure X cascade into failure Y?

Collect the rebuttals. They surface blind spots, expose echo-chamber findings, and reveal cascading chains a single-pass fan-out misses.

If `--fast` is set: skip this phase entirely. The fast mode is for cheap reconnaissance, not depth.

### Phase 4 — Synthesis (main thread)

Now collapse everything into a coherent picture.

1. **Deduplicate** failure modes across personas into families.
2. **Classify** each family using `references/classification.md`:
   - **Confirmed** — evidence-backed real risk
   - **Inflated** — sounds scary on first read but downgrades on inspection (low likelihood or low impact)
   - **Buried** — politically unspoken; no one wants to name it
3. **Assign urgency** to each Confirmed risk:
   - **Block** — must resolve before proceeding
   - **Patch-fast** — must address within the first window post-launch
   - **Monitor** — watch but don't gate
4. **Score** each Confirmed risk on the 6 severity lenses and likelihood.
5. **Pick the top 5–7** risks. For each, write:
   - Mitigation (concrete action available this week)
   - Tripwire (the field signal to watch for)
   - Owner shape (the type of role that would own this — not a specific person)
   - "What would change my mind"
6. **Compute revised confidence** using the math in `references/synthesis.md`:
   - Unmitigated success probability (X%)
   - With top-3 mitigations applied (Y%)
7. **Identify the most important Buried risk** for the TL;DR.
8. **Produce the revised plan**: take the original plan and fold the mitigations in inline as new bullets, sub-steps, or annotations. Don't rewrite tone — just inject.

### Phase 5 — Write the artifacts

Determine the output directory:
- `--output` flag, OR
- `$TIME_TRAVEL_REPORTS_DIR` env var, OR
- default: `./time-travel-reports/` (CWD)

Compute a timestamp + slug: `YYYY-MM-DD-HHMM-<short-slug>`.

Always write the report. Use `references/report-template.md`:
- `<timestamp>-<slug>-report.md`

Unless `--no-process-log`, also write the process log. Use `references/process-log-template.md`:
- `<timestamp>-<slug>-process.md`

If `--for-exec`, also write the exec one-pager. Use `references/exec-one-pager-template.md`:
- `<timestamp>-<slug>-exec.md` — strict ≤300-word limit. Mechanically count words before writing.

Print to the chat:
- TL;DR section (top 3 failures + revised confidence + Buried risk)
- Paths to each artifact

## Anti-patterns (read before you start)

See `references/anti-patterns.md` for the full list. The most common failures of premortem skills:

- Producing generic, plan-agnostic risks ("scope creep", "team burnout") that could apply to anything.
- Treating technical severity as the only severity. Trust damage, user pain, and ops load matter equally.
- Forgetting the **Buried** category — every plan has politically unspoken risks; surfacing zero is a process failure, not a clean bill of health.
- Skipping the tripwire — a risk without a field signal is unactionable.
- Skipping the "What would change my mind" — turns the report into criticism the user can dismiss.
- Letting all personas agree. If the rebuttal round produces no disagreement, the personas are too similar — rerun with a more diverse roster.
- Listing too many Confirmed risks (>7). If everything is critical, nothing is.
- Writing the revised plan as a wholesale rewrite. Inject mitigations into the original structure; don't replace it.

## Tools used

- `WebSearch` — for the evidence pack in Phase 2 and follow-up grounding in Phase 4.
- `Agent` (via `superpowers:dispatching-parallel-agents`) — for the persona fan-out in Phase 3 and the rebuttal round in Phase 3.5.
- `Write` — for the final artifacts in Phase 5.
- `Read` — for the plan source in Phase 0.
